Overview

Built during my tenure as Software Engineer at East Anglia Digital Solutions, this project involved designing and engineering a high-throughput, microservices-based API gateway to act as a single, centralized entry point for all incoming service requests — handling routing, authentication, traffic shaping, and failure isolation across a growing fleet of independent backend services.

As the company's product suite expanded into a sprawling microservices ecosystem, the absence of a unifying gateway layer had become one of the biggest sources of operational risk in the stack. The goal wasn't simply a reverse proxy — it was a well-instrumented choke point through which all traffic could be observed, secured, and controlled.

The Problem

  • Inconsistent authentication — each service implemented its own auth logic, leading to security gaps and duplicated code across the platform.
  • No centralized traffic visibility — no single place to monitor request volume, latency, or error rates.
  • Cascading failures — a slow or failed downstream service caused client requests to hang across unrelated parts of the system.
  • Redundant network overhead — repeated calls for the same data hit backend services directly with no caching layer in front.

Approach & Implementation

The gateway was designed around four key components:

  • Asynchronous request handling — built on a non-blocking, event-driven I/O model so a single instance could process thousands of concurrent requests without threads idling on slow downstream services. Connection pooling was tuned per service to prevent one spike from exhausting shared resources.
  • Localized caching — a caching layer intercepted frequently requested, low-volatility data before it reached downstream services. Cache keys were derived from request path and query parameters, with TTLs configured per endpoint based on data volatility, agreed in collaboration with each owning team.
  • Centralized token-based authentication — the gateway became the single authority for verifying JWT access tokens (signature, expiry, scope/permission checks), so backend services could trust an internal header instead of re-implementing auth logic.
  • Failure isolation & resilience — per-service timeouts and circuit breakers automatically pulled a struggling downstream service out of rotation, paired with structured error responses and centralized logging for fast root-cause visibility.

Key Design Decisions & Challenges

  • Cache invalidation correctness — deciding which endpoints were safe to cache, and for how long, required close coordination with downstream teams to avoid serving stale or sensitive data.
  • Backward compatibility — the gateway was introduced without breaking existing client integrations, supporting legacy auth headers during a phased migration.
  • Balancing centralization with autonomy — the gateway was kept thin and limited to cross-cutting concerns, so it didn't become a bottleneck for feature development by individual service teams.

Outcome

The result was a markedly more resilient system with reduced latency on high-traffic, read-heavy endpoints (via the cache-hit fast path) and a single, well-instrumented point of control for security and monitoring. Platform uptime improved because failures in one service no longer cascaded into others — the gateway could detect and isolate a struggling service before the problem spread. The team gained a consistent place to reason about authentication, traffic patterns, and failure behavior across the entire backend.

Key Takeaways

Reduced average latency on cached, read-heavy endpoints via async I/O and localized caching
Centralized authentication eliminated duplicated, inconsistent auth logic across services
Improved platform resilience by isolating service-level failures with timeouts and circuit breakers
Established a single, centralized point of observability for backend traffic and incident response

Overview

Built during my time as a Game Developer at Interactive Pixel Studios (2019–2022), this project involved programming a robust, component-based gameplay ability and interaction system in C#, built around abstract finite state machines (FSMs) — in response to a recurring need across multiple titles for a flexible way to define complex player behaviors, enemy AI responses, and modular combat interactions without hardcoding rigid, hard-to-extend logic.

Earlier projects at the studio had relied on large, branching conditional blocks to manage character behavior — a pattern that became increasingly fragile and difficult to extend as gameplay complexity grew. This system replaced that approach with something structured, reusable, and safe to extend without breaking existing behavior.

The Problem

  • Monolithic conditional logic — enemy AI and player ability behavior were often implemented as large if/else or switch blocks within a single script, making it hard to reason about all possible character states.
  • Fragile extension — adding a new behavior frequently risked breaking unrelated existing behavior, since logic for different states lived intertwined in the same methods.
  • Poor reusability across characters — behavior logic was typically written per-character rather than as shared, composable building blocks, leading to duplicated code across enemy types and player abilities.
  • Slow prototyping — designers and programmers had to make invasive code changes for even small behavioral tweaks, slowing iteration during combat design.

Finite State Machine Design

At the core of the system is an abstract state machine pattern: each behavioral state (Idle, Patrol, Chase, Attack, Stagger, Dead, and so on) is implemented as its own self-contained class with Enter, Update, and Exit methods, rather than as a branch inside a larger function. Transitions between states are defined as explicit, named rules evaluated each frame, rather than buried inside conditional logic.

Because each state is fully self-contained, adding a new behavior — such as a ranged attack state or a fleeing state — meant writing one new class and wiring up its transition conditions, without modifying any existing states.

Component Architecture

The state machine was built as a generic, reusable component rather than something hardcoded into any one character type. A central State Machine Controller managed the current active state and evaluated transition rules each frame, while concrete state implementations and their transition conditions were swappable per character — the same underlying FSM engine powered enemy AI, player ability sequencing, multi-hit combo systems, and interactable objects (levers, doors, puzzle triggers).

Approach & Implementation

  • Abstract state base class — defined the contract every state had to fulfill (Enter, Update, Exit), so any programmer on the team could read or extend a state without understanding the entire system first.
  • Data-driven transition rules — transitions were defined as discrete condition objects (e.g. "player spotted," "health reaches zero") evaluated each frame, making the behavior graph easy to visualize and modify without touching internal state logic.
  • Reuse across gameplay systems — the same FSM component powered player ability sequencing (windup, active, recovery, cooldown) and multi-hit combat combos, rather than building separate one-off systems for each.
  • Rapid prototyping support — state and transition definitions were exposed through the Unity Inspector wherever practical, letting designers compose new behaviors without requiring a programmer for every experimental tweak.

Challenges & Design Decisions

  • State explosion — managed by introducing optional hierarchical sub-states for complex behaviors (e.g. nesting attack sub-states within a broader "Attack" state) rather than flattening everything into the top-level machine.
  • Performance with many active agents — transition conditions were batched and update frequency staggered across agents to avoid all FSMs evaluating in the same frame.
  • Debuggability — a runtime state visualizer was added so designers and programmers could see the current state and recent transition history for any active agent during play, which proved essential for diagnosing unexpected behavior.

Outcome

The system enabled quick prototyping of complex player behaviors, enemy AI responses, and modular combat interactions, replacing fragile, hard-to-extend conditional logic with a structured, predictable, and visually traceable behavior model. Because the core FSM engine was built to be generic rather than single-purpose, it became a shared foundation reused across enemy AI, player ability systems, and combat combo design — reducing duplicated effort and giving the design team a consistent mental model for character behavior across the entire game.

Key Takeaways

Replaced fragile, monolithic conditional logic with a structured abstract state machine pattern
Built a single, generic FSM engine reused across enemy AI, player abilities, and combat combos
Enabled new states and transitions to be added without modifying existing behavior
Supported rapid, designer-led prototyping of complex player and enemy behaviors

Overview

Carried out during my time as a freelance AI Data & Technical Research Contributor (2017–2019), this project involved curating, structuring, and auditing extensive text and image datasets intended for supervised machine learning applications, working with teams that needed clean, reliable training data to build accurate downstream models.

Rather than treating dataset preparation as a one-off cleanup task, I approached it as a repeatable pipeline — a structured sequence of curation, annotation, and validation stages, each with its own quality checks, so data quality could be measured and improved systematically rather than left to chance.

The Problem

  • High label noise — a meaningful percentage of labels in raw datasets were incorrect, inconsistent, or ambiguous, directly degrading model accuracy regardless of model tuning.
  • No systematic quality measurement — no consistent way to quantify how "clean" a dataset actually was before training began, so problems were often only discovered after a model had already underperformed.
  • Duplicate and near-duplicate samples — raw datasets, especially scraped or aggregated ones, frequently contained redundant entries that skewed class balance and inflated apparent dataset size without adding real signal.
  • Inconsistent annotator judgment — where multiple annotators labeled similar data, there was often no process for measuring or reconciling disagreement, allowing subjective inconsistencies to silently enter the training set.

Pipeline Architecture

The pipeline was structured as five sequential stages, with a continuous evaluation layer feeding back into the annotation stage whenever validation flagged problematic data. This feedback loop was the key structural decision: rather than treating validation as a final gate, it was used to actively drive relabeling and correction throughout the process.

Approach & Implementation

  • Curation & deduplication — raw text and image datasets were passed through deduplication logic using similarity hashing and embedding-based nearest-neighbor comparisons to remove exact and near-duplicate samples, meaningfully reducing apparent label noise from redundant, inconsistently-labeled entries.
  • Structured annotation — annotation guidelines were written to be explicit and example-driven rather than relying on subjective interpretation, with ambiguous edge cases collected into a reference set so annotators had consistent precedent to follow.
  • Custom evaluation metrics — developed metrics specifically for measuring dataset quality before it reached a model, including inter-annotator agreement scoring, confidence scoring on individual labels, and statistical outlier/anomaly detection.
  • Data validation workflows — flagged samples were routed back into a relabeling workflow rather than simply discarded, preserving as much usable data as possible while correcting quality issues.

Challenges & Design Decisions

  • Balancing recall vs. precision in noise detection — overly aggressive outlier flagging risked discarding valid but rare examples; thresholds were tuned conservatively and paired with human review rather than automatic removal.
  • Annotator guideline drift — as edge cases accumulated, guidelines needed periodic revision; a versioned guideline document with changelog was maintained so it was always clear which rules applied to which labeling batch.
  • Scalability of manual review — flagged-sample volume was prioritized by estimated impact on model performance so review effort was spent where it mattered most.

Outcome

The custom evaluation metrics and validation workflows lowered label noise across the datasets processed, directly improving final model accuracy scores on downstream tasks. On a representative project, estimated label noise was reduced from roughly 18.5% in the raw dataset to approximately 3.1% after the full pipeline — translating into measurably better model performance without changing model architecture or training procedure. Beyond the immediate quality gains, the pipeline gave the teams a repeatable, measurable process for dataset preparation, rather than ad hoc cleanup performed differently on every project.

Key Takeaways

Built a repeatable five-stage pipeline for curating, annotating, and validating ML training data
Developed custom evaluation metrics including inter-annotator agreement and outlier detection
Reduced estimated label noise from ~18.5% to ~3.1% on a representative dataset
Introduced a feedback loop routing flagged data back into relabeling rather than discarding it

Overview

Carried out during my time as a freelance AI Data & Technical Research Contributor (2017–2019), this project involved establishing structured evaluation metrics and edge-case prompt libraries to benchmark deep learning models, with a particular focus on generative AI systems — working on red-teaming and benchmarking protocols intended to surface model weaknesses before they reached production or downstream fine-tuning.

The work centered on a demanding goal: stress-test generative models against deliberately difficult, adversarial, and edge-case inputs, then translate the resulting failures into structured, actionable data that could be fed back into reinforcement learning fine-tuning to make the model genuinely more robust, not just superficially better on average-case inputs.

The Problem

  • Shallow evaluation coverage — typical test sets skewed toward common, easy queries, leaving models under-tested on ambiguous instructions, conflicting constraints, and deliberately adversarial prompts.
  • Unstructured failure reporting — no consistent framework for categorizing why a model failed (factual inaccuracy, hallucination, constraint violation, instruction-following failure), making it hard to prioritize fixes.
  • No reusable adversarial prompt set — red-teaming was often performed ad hoc, with prompts invented on the fly rather than maintained as a structured, growing library.
  • Disconnection from fine-tuning — evaluation results often stayed as static reports rather than being converted into structured signal usable for RL fine-tuning.

Protocol Architecture

The protocol was structured as a closed loop, not a one-time test. A maintained library of edge-case prompts was run against the model under test, outputs were scored across three core evaluation dimensions, results were converted into structured, severity-tagged annotations, and those annotations ultimately fed into a reinforcement learning fine-tuning dataset — closing the loop between evaluation and actual model improvement.

Approach & Implementation

  • Edge-case prompt library — built and maintained a structured library of adversarial prompts probing known failure modes (ambiguous instructions, policy-boundary requests, technically dense factual queries, hallucination-inducing prompts), organized by category and severity.
  • Structured evaluation metrics — established three distinct evaluation dimensions scored separately: technical accuracy, constraint adherence, and hallucination reduction, making it possible to see precisely where a model was weak rather than collapsing everything into one score.
  • Red-teaming rounds — evaluation ran in successive rounds, with the lowest-scoring categories used to expand the prompt library with harder, more targeted variants after each round — the evaluation set evolving to stay ahead of the model's current weaknesses.
  • Structured scoring & RL fine-tuning handoff — every scored output was tagged with a severity tier and failure category, giving the model team concrete examples of failure paired with severity-weighted signal rather than vague qualitative feedback.

Results

A radar chart comparing benchmark scores across all five evaluation dimensions between the first red-teaming round (baseline) and the fourth round showed the largest gains in hallucination resistance and edge-case robustness — precisely the dimensions the protocol was designed to target — consistent with the evaluation set successfully surfacing and helping correct the model's weakest areas.

Challenges & Design Decisions

  • Avoiding evaluator bias — scoring rubrics were written with explicit, example-anchored criteria for each severity tier to reduce subjective drift between evaluation rounds and evaluators.
  • Preventing prompt library staleness — once a model learned to handle a given prompt, its diagnostic value dropped; the library was actively curated, with solved cases retired in favor of harder variants.
  • Balancing severity weighting — severity tiers were calibrated so high-stakes failures (e.g. confident factual hallucination) carried more weight in the RL signal than minor stylistic issues.

Outcome

The structured evaluation metrics and edge-case prompt libraries successfully benchmarked the models under test across technical accuracy, constraint adherence, and hallucination reduction, with resulting structured scores providing the critical data needed for RL fine-tuning. Across four red-teaming rounds, measured scores improved substantially across every evaluation dimension, with the most significant gains concentrated in the areas the protocol was specifically designed to probe. The protocol itself became a reusable asset — a structured, evolving prompt library and scoring framework applicable to future model versions rather than rebuilding red-teaming infrastructure from scratch each cycle.

Key Takeaways

Built and maintained a structured, evolving library of adversarial and edge-case prompts
Established three evaluation dimensions: technical accuracy, constraint adherence, hallucination reduction
Ran iterative red-teaming rounds progressively targeting the model's weakest areas
Converted evaluation results into structured, severity-tagged data usable for RL fine-tuning

Overview

Developed during my time as a Junior Software Developer at Suffolk Technology Services (2016–2019), this project involved creating an internal desktop application to automatically aggregate and parse complex system logs across local development environments — built in response to the engineering team spending a disproportionate amount of time manually digging through scattered log files to identify the root cause of recurring issues.

Rather than another one-off debugging script, the goal was a standing internal tool the whole engineering team could run locally, point at their development environment, and get back structured, readable error trends instead of raw, scattered log text.

The Problem

  • Fragmented log sources — logs were scattered across multiple local services and environments, with no single place to view them together, forcing engineers to manually cross-reference several files for a single issue.
  • Unstructured, noisy raw text — raw log output mixed critical errors with routine informational noise, making it genuinely difficult to spot the signal that mattered, especially under time pressure.
  • No visibility into error frequency or trends — no easy way to tell whether an error was a one-off or part of a recurring pattern, meaning the same root causes were often rediscovered repeatedly by different engineers.
  • Time lost to repetitive manual triage — the cumulative time spent simply locating and reading relevant log lines was a significant and avoidable drag on engineering productivity.

System Architecture

The application was structured as a three-stage local pipeline: log aggregation and parsing, custom rule-based filtering, and a diagnosis dashboard that presented results as readable error trends rather than raw text — taking logs from multiple local development environments and turning them into a single, structured view.

Approach & Implementation

  • Log aggregation & parsing — the application watched and ingested log output from multiple local development environments simultaneously, parsing each line into a structured format (timestamp, source, severity, message) — the foundation that made every later filtering and aggregation step possible.
  • Custom filtering rules — built a filtering rules engine allowing engineers to define and save filters based on severity, source, keyword patterns, and frequency thresholds, suppressing noisy informational logs by default while keeping critical errors visible.
  • Error trend aggregation — the tool grouped recurring errors together and tracked their frequency over time, directly solving the "is this a one-off or a pattern" problem and surfacing which issues were most worth prioritizing.
  • Diagnosis dashboard — a desktop dashboard presenting aggregated, filtered results in a readable interface: a summary view of top recurring errors, a searchable detail view, and visual indicators of error frequency over time.

Challenges & Design Decisions

  • Handling inconsistent log formats — different services logged in slightly different formats; a flexible, configurable parsing layer was built rather than hardcoding a single expected format, so new log sources could be added without rewriting the parser.
  • Avoiding filter over-suppression — overly aggressive default filtering risked hiding genuinely important but infrequent errors; defaults were deliberately conservative, with engineers opting into stricter filtering rather than the tool silently hiding rare but critical issues.
  • Keeping it lightweight — since this ran locally alongside active development environments, the tool was built to be resource-light, avoiding heavyweight features that would compete with the engineer's actual workflow for system resources.

Outcome

Custom filtering rules turned raw, scattered log text into readable error trends, slicing debugging times down significantly for the internal engineering team. Based on tracked usage during rollout, total time to diagnose a typical issue dropped from roughly 95 minutes to approximately 28 minutes on average, with the largest gains coming from faster root-cause location and reduced time spent manually cross-referencing multiple log sources. Beyond the direct time savings, the tool gave the team a shared, consistent way to talk about recurring issues — referencing a named error trend in the dashboard rather than describing a vague, hard-to-reproduce symptom.

Key Takeaways

Built a desktop tool aggregating and parsing logs across multiple local development environments
Designed a custom, configurable filtering rules engine to separate signal from noise
Introduced error trend aggregation to surface recurring patterns rather than isolated incidents
Reduced average issue diagnosis time from approximately 95 to 28 minutes
